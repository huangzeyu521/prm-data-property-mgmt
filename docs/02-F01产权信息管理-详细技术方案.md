# F-01 产权信息管理 — 详细技术方案（严格信创栈）

> 范围：数据产权管理模块第一期 **F-01 产权信息管理**（2 子项 / 12 主界面）。
> 技术路线：严格信创栈（SpringCloud+TSF / 达梦 DM8 / 国密 / Camunda / 银河麒麟 / 鲲鹏）。
> 前置基线：见 `01-数据产权管理-需求与设计综述.md`。本文为开发前的技术方案/计划，暂不含业务代码。
> 版本：v0.1（待评审）。日期：2026-06-01。

---

## 1. 范围与目标

### 1.1 功能范围（F-01，界面前缀 `IM-DAM-DPR-01`）

| 子项 | 页面（主界面编码） |
|---|---|
| **产权信息台账** (01-001-001) | 产权总体概览(001)、数据资产产权树展示(002)、数据集产权档案管理(003)、数据集详情展示(004)、产权信息查询筛选(005)、产权变更记录管理(006)、产权台账统计分析(007) |
| **权益动态监测管理** (01-001-002) | 权益状态监控(001)、权益变动监测预警(002)、合规性检查(003)、监测规则配置(004)、风险预警通知(005) |

### 1.2 F-01 在业务链路中的定位

F-01 是整个模块的**数据底座 + 风控中枢**：
- **台账**：消费数据资产管理平台的资产目录/认责信息，建立"一数一档"产权档案，被动接收确权(F-02)/授权(F-03)审批流的权属变更事件并落档。
- **动态监测**：基于规则引擎对权益卡片/授权状态做实时监测、合规检查、风险预警闭环。

> 设计取舍：F-01 自身**不产生权属（不审批）**，权属由 F-02/F-03 产生；F-01 通过**事件订阅**持久化变更、通过**规则引擎**做监测。这与需规"变更记录由确权/授权审批流自动触发""新增为事件驱动"一致。因此 F-01 可在 F-02/F-03 尚未完成时，先以**资产目录同步 + 手工/接口灌入**方式独立建设与联调。

## 2. 总体技术架构

### 2.1 分层（信创栈）

```
┌───────────────────────────────────────────────────────────┐
│ 接入层  统一服务网关(Spring Cloud Gateway/TSF) · mTLS · 4A SSO │
├───────────────────────────────────────────────────────────┤
│ 前端    Web(数研院规范, Vue) · 南网eLink(H5/小程序, 仅预警通知) │
├───────────────────────────────────────────────────────────┤
│ 应用层  dpr-ledger-service(产权信息台账)                       │
│         dpr-monitor-service(权益动态监测/规则/合规/预警)        │
│         dpr-common(鉴权/字典/审计/文件/导出 公共能力)           │
├───────────────────────────────────────────────────────────┤
│ 能力层  Camunda(预警处置流) · Drools(监测/合规规则引擎)         │
│         国密SDK(SM2/3/4) · 导出引擎(Excel/PDF)                 │
├───────────────────────────────────────────────────────────┤
│ 集成层  catalog-sync · steward-query · lifecycle-consumer     │
│         change-event-consumer(订阅F-02/F-03) · 4A-client      │
│         elink-push · minio-client                             │
├───────────────────────────────────────────────────────────┤
│ 数据层  达梦DM8(主备) · Redis(缓存/分布式Session) ·            │
│         Kafka(事件) · MinIO(附件/快照) · (Neo4j 可选, 血缘)     │
└───────────────────────────────────────────────────────────┘
```

### 2.2 微服务划分（建议两个业务微服务 + 一个公共库）

| 服务 | 职责 | 主要表 |
|---|---|---|
| `dpr-ledger-service` | 产权档案 CRUD、产权树、查询筛选、变更记录、台账统计、概览 | DATA_ASSET_INFO、PROPERTY_ARCHIVE、CHANGE_RECORD |
| `dpr-monitor-service` | 权益状态监控、变动预警、合规检查、监测规则、预警通知 | PERMISSION_DYNAMIC、MONITOR_RULE、COMPLIANCE_RESULT、ALERT_RECORD |
| `dpr-common`（依赖库/或独立 starter） | 4A 鉴权过滤器、数据范围(ABAC)切面、字典、审计留痕、文件、导出、国密工具 | DICT、AUDIT_LOG |

> 注册中心 Consul/Eureka；配置中心 TSF 配置；服务间 OpenFeign + mTLS。可按需先合并为单体 `dpr-service`（模块化包结构）再拆分，降低首期复杂度——见 §10 里程碑。

## 5. 关键流程设计

1. **资产目录同步 → 产权树/档案底座**：定时(全量)+增量+事件三通道；建 `资产ID映射表` 防"幽灵资产"；百万级用断点续传+批处理；`pra_dam_steward_query` 补全责任主体（Redis 缓存）。
2. **变更记录自动生成（防篡改）**：监听 F-02/F-03 审批流 Kafka 事件 → 写 CHANGE_RECORD（前后值快照）→ 调区块链上链取 SM3 回执 → 人工增删改走高权限 RBAC + 特权操作日志。
3. **监测规则引擎**：规则在 MONITOR_RULE（DRL/JSON）→ 发布前沙箱回归测试 → 注册到 Drools；生效规则禁物理删，仅停用 + 新版本。
4. **权益状态监控 + 合规巡检**：后台巡检引擎(定时/实时)比对 权益卡片有效期 + 网关调用日志 + 授权边界 → 命中规则写 COMPLIANCE_RESULT / 触发 ALERT_RECORD。
5. **风险预警闭环**：ALERT_RECORD 自动定级(红/黄/绿) → eLink/站内消息推送责任人/合规小组 → 处置(核实/整改/移交)→ 超时自动督办升级 → 合规小组结单闭环；统计"按时响应率/整改闭环率"作为 KPI。预警记录禁物理删，仅标记 已处理/已忽略/误报。

## 6. 前端方案（Vue，遵循数研院规范 + "高压线"）

- **工程**：Vue 全家桶（路由/状态/请求封装）+ 数研院 WEB UI 组件（.xd/.rplib 转译为统一组件库）；ECharts 图表；按需国密浏览器插件。
- **路由**：`/dpr/ledger/*`、`/dpr/monitor/*`，12 页面一一对应界面编码。
- **强制规范**：必填红星居左；提示分浮动(正中偏上)/对话框(正中央)两类、中文半角无感叹号；按钮主蓝次白、命名用统一表(查询/重置/新增/导入/导出/保存/提交/启用/停用…)；表格首列序号、行≤3 行超出"…"悬浮；弹窗居中屏蔽背景；返回保留查询条件/页码；查询框过滤首尾空格；微软雅黑、≥12px。
- **公共组件**：产权树、档案详情侧滑、前后值红绿对比、风险等级徽章、规则配置器、导出按钮、看板卡片组。
- **eLink**：仅"风险预警通知"做 H5 待办/推送。

## 7. 安全与权限落地

- **鉴权**：网关 4A SSO(OAuth2.0/OIDC) + Access Token；Redis 分布式 Session；敏感操作(删除档案/处置预警)MFA。
- **数据范围 ABAC**：`dpr-common` 切面按 4A 组织代码 + 敏感度 + 业务条线注入 SQL 过滤（数据不出域）；仅总部数字化部主管有例外手工修正权。
- **国密**：敏感字段 SM4 落库；变更/日志 SM3 完整性 + HMAC；传输 TLS/国密。
- **审计**：所有写操作记 AUDIT_LOG（毫秒级时间、4A 账号、源 IP、对象、结果），在线≥6 个月，上报安全管理中心；特权操作单独留痕。

---

## 8. 非功能落地要点

并发≥100/TPS≥50/简单≤3s 复杂≤5s（缓存+分页+索引+异步导出）；可用性≥99.9%（DM8 主备+多实例+熔断降级）；等保三级；信创全栈适配（DM8 方言、麒麟、鲲鹏、TongWeb/Apusic、国密、信创浏览器）。

## 9. 风险与待确认事项

1. **新增表编码对齐**：§3.2 五张★表需与数据架构组对齐 `IM_*`/`CEC_*` 正式编码与逻辑模型（可研/数据架构 v12 仅显式列出 2 张）。
2. **F-02/F-03 事件契约**：变更/权限动态依赖确权、授权审批流的 Kafka 事件 schema —— F-01 先期需定义"事件契约草案"，或以接口/手工灌入做联调桩。
3. **合规检查数据源**："网关调用日志"来源（IOS/全链路监控？）需确认对接方与字段。
4. **达梦/Camunda 适配**：Camunda 在 DM8 上的建表脚本需验证；优先用于"预警处置流"，台账类不引入流程引擎。
5. **单体 vs 微服务**：建议首期单体模块化(`dpr-service`)，跑通后再拆 ledger/monitor，降低信创环境部署成本。
