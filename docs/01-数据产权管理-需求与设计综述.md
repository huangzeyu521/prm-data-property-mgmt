# 数据产权管理模块 — 需求与设计综述

> 本文档是对 `materials/` 全部资料研读后的综合提炼，作为后续设计与开发的统一基线。
> 来源：数据资产管理平台 V3.6 建设项目（中国南方电网 / 云化数据中心）。
> 编制日期：2026-06-01。

---

## 0. 一句话定位

**数据产权管理**是数据资产管理平台 V3.6 中"数据流通管理"下的**一个二级应用模块**（模块编码 M-DP-01 / 界面前缀 `IM-DAM-DPR`），围绕国家"数据二十条"**三权分置**（数据资源持有权、数据加工使用权、数据产品经营权），实现数据资产的**确权—授权—权益监测—分析**全流程线上闭环。建设模式："总部统一建设，全网推广使用"。核心理念：**"只填一张表" + "先确后授、有权才用"**。

---

## 1. 项目背景与目标

- **背景痛点**：① 确权/授权全流程缺乏线上管控（多部门、多凭证、人工评审，周期长、易缺材料、无留痕）；② 产权信息未流通共享，存在"数据黑箱"；③ 2000+ 业务系统、百万级数据表，人工分类不可持续，需 AI 辅助。
- **核心目标**：流程线上化 + 基层减负（自动引用资产/元数据，"只填一张表"）；智能审核（AI 大模型辅助材料解析、合规校验、权属冲突识别）；合规可控（表级权属、强制"先确后授"）；可溯源 + 存证（全链路留痕 + 区块链存证，形成权利链）。
- **试点路径**：约项目第 8 个月在试点单位（广东电网或总部）对高频表（用户信息表、设备台账表）以"AI 辅助 + 人工复核"方式试点"只填一张表"，再全网推广。

## 3. 核心业务规则（贯穿性、强约束）

1. **先确后授**：未确权数据不可授权；系统硬性拦截。
2. **授权不得超确权边界**：下游授权的范围/字段/时效/地域必须 ⊆ 上游权益卡片确权边界；超范围/超期红字拦截。
3. **三权分置**：持有权 / 加工使用权 / 产品经营权，分别确权、分别发证。
4. **两种授权模式**：批量授权（年度集中）+ 一事一议专项授权（按需）。
5. **审批流（BPM，Camunda）**：
   - 确权：归集审查(40) → 合规审查(50) → 主管复核(60) → 领导终审(70) → 制作权益卡片/归集(80)；审批深度按数据敏感级动态调整；驳回支持"退经办人补正 / 退上一级重审"。
   - 专项授权：业务部门初审 → 数字化部复核 → 合规小组审查 → 总部数字化部审批。
   - 批量授权：数字化部归集 → 合规小组审核 → 总部数字化部认定 → 领导小组办公室批准。
6. **防篡改 / 不可抵赖**：进度记录、变更记录、审核意见、权益卡片、证书均由流程引擎事件回调**自动生成**；人工增删改需高权限 RBAC + 特权操作日志；生效规则、预警记录、已用模板、已归档协议**禁止物理删除**（仅停用/作废/新版本/状态标记）。
7. **存证**：确权结果、授权协议、关键操作日志区块链上链存证。
8. **数据隔离**：到处都是 RBAC + 组织层级范围控制（数据不出域）；仅总部数字化部**主管**对基础台账/变更记录有例外手工修正权。
9. **唯一编码**：证书编号全局唯一（雪花算法 / UUID）；卡片↔证书↔审批结果 1:1 对应。

## 4. 数据模型（物理表，前缀 `IM_` / `IM_DPM_`，字段前缀 `CEC_`）

数据域=数字化域(DA-SZ)，子域=数据管理(DA-SZ-DS46)；安全等级一级；均含 `PROVINCE_CODE/BUREAU_CODE/CREATE_TIME/CREATOR_ID`。

| `IM_CONFIRM_LIST_INFO` | 确权清单信息表 | CEC_CONFIRMATION_PROJECT_ID(PK), PROJECT/TIME/RIGHTS_HOLDER/STATUS/RESULT |

| `IM_PRA_MATERIAL_CHECK_RESULT` | 材料校验结果信息表 | CEC_MATERIAL_NO(PK), VERIFY_RESULT/TIME, ABNORMAL_DESC |
| `IM_AUTH_REQ_INFO` | 授权需求信息表 | CEC_AUTH_REQUIREMENT_TYPE_ID(PK), APPLICANT/TYPE |

关键状态枚举：申请状态(草稿/审批中/已归档)、确权状态(生效/已撤销/过期)、权益状态(正常/冻结/失效)、协议状态(生效中/已终止/归档)、权限变动类型(新增授权/到期回收/风控熔断)。

## 5. 集成接口（RESTful + Kafka 事件，统一网关 mTLS）

### 5.1 与人工智能平台（大瓦特）
| 服务代码 | 名称 | 说明 |
|---|---|---|
| pra_ai_ocr (I-PRA-001) | OCR 文档智能解析 | 提取权利主体/客体/类型/有效期，准确率≥95% |
| pra_ai_conflict (I-PRA-002) | 权属冲突识别 | 主体/范围/时效/历史记录四类冲突 + 风险评分（AUC≥0.90） |
| pra_ai_decision (I-PRA-003) | RAG 确权决策预测 | 建议通过/驳回/补充材料 + 因子权重(SHAP) |

算法栈（由 AI 平台维护）：OCR=ResNet50+CTC+BiLSTM+BERT；冲突=GraphSAGE+XGBoost；决策=BGE-large-zh 召回 + 大瓦特 L0 大模型 + RAG。响应≤5s，可用性≥99.9%。

### 5.2 与数据资产管理平台同平台模块（7 个）
| 服务代码 | 名称 | 来源模块 |
|---|---|---|
| pra_dam_catalog_sync (I-PRA-DAM-01) | 数据资产目录同步 | 数据目录管理（全量/增量/事件） |
| pra_dam_quality_check (I-PRA-DAM-02) | 数据质量校验 | 数据质量管理（评分<80 自动驳回确权） |
| pra_dam_steward_query (I-PRA-DAM-03) | 数据认责查询 | 数据认责管理（Redis 4h 缓存） |
| pra_dam_demand_relay (I-PRA-DAM-04) | 数据需求授权申请透传 | 数据需求管理（启动授权工作流） |
| pra_dam_lifecycle_sub (I-PRA-DAM-05) | 生命周期事件订阅 | 数据生命周期管理（Kafka，归档/销毁/迁移→权益卡片失效，FSM） |
| pra_dam_card_writeback (I-PRA-DAM-06) | 权益卡片回写 | 数据目录管理（最终一致+乐观锁 version，指数退避重试） |
| pra_dam_share_check (I-PRA-DAM-07) | 授权有效性核验 | 数据对内共享管理（先确后授校验） |

### 5.3 平台级集成
- **4A / 数字身份与访问管理平台**：不建本地账号；OAuth2.0/OIDC SSO；账号/组织单向同步(JADP/LDAP/SCIM)；RBAC+ABAC；敏感操作 MFA。
- **统一密码服务平台（国密）**：SM2 身份鉴别、SM4 字段加密、SM3 完整性、HMAC 日志防篡改、电子签章 + CA 核验。
- **IOS 信息安全运行监测预警系统**：资产纳管、主机安全(G01)、运维监控、日志全量上报(Syslog/Kafka)。
- **南网 eLink**：移动端待办/审批/通知推送、证书数字大厅出示。
- **非结构化数据服务平台（MinIO/对象存储）**：材料、协议、证书模板底图/水印。
- **区块链平台**：确权结果/协议/关键日志上链存证。

## 6. 非功能需求（关键指标）

- **性能**：注册用户≥2000，最大在线≥200，并发≥100，TPS≥50；登录与简单操作≤3s，复杂≤5s，AI/上链异步≤8s；CPU<65%，内存<70%。
- **可靠性**：可用性≥99.9%；MTBF>2000h；MTTR<30min（一般）/<4h（严重）；7×24。
- **灾备**：核心确权/授权 RPO<30min，一般日志<24h；关键业务 RTO<4h；同城双活灾备 5 级（Multi-AZ，RPO=0 数据库 Data Watch）；周备六、日增量、季度演练。
- **安全**：等保三级；数据三级分类（核心商密/普通商密/工作秘密）；三权分立（系统/安全/审计管理员）；5 次失败锁定、15min 超时登出；日志在线≥6 个月。
- **兼容/自主可控（信创）**：CPU=鲲鹏/飞腾/龙芯/海光；OS=银河麒麟 V10/统信 UOS；DB=达梦 DM8（备：人大金仓）；中间件=东方通/金蝶/中创；浏览器=奇安信/360/统信等 + Chrome72+；微服务=SpringCloud + 腾讯云 TSF；注册中心 Consul/Eureka；消息 RabbitMQ/Kafka；图库 Neo4j。

## 7. 前端界面规范（强制"高压线"，建设必须遵守）

- **页面结构**：同级元素间距统一；分组对齐清晰；表单文字标签右对齐、输入框左对齐；工作区最大化、无多余滚动条、无过多留白。
- **必填符号**：红色星号 `*` 固定在文字标签**左侧**；查询条件**不得**带星号。
- **提示信息**：① 浮动提示（轻量反馈，屏幕正中偏上，成功/风险警告）；② 对话框提示（二次确认，屏幕正中央，删除等）；措辞简洁、中文半角标点、**避免感叹号**、避免技术细节。
- **按钮**：主按钮蓝色、次按钮白底黑字；按钮集最重要在最左、删除居中；全系统统一一套组件；命名遵循统一表（查询/重置/新增/导入/导出/保存/提交/审批/回退/撤回/确定/取消/启用/停用…）。
- **表格**：第一列为序号；无多余选框；行内最多 3 行，超出"…"悬浮显示全文。
- **字体**：中文微软雅黑、英文 Arial；最小字号≥12px。
- **交互**：弹窗居中且屏蔽背景、无多余关闭按钮；返回保留查询条件/排序/页码；查询框过滤首尾空格；鼠标指针随语义变化；浮出层置顶不被遮挡。
- **设计基调**：参见《设计原则及典型界面示例》《WEB 用户界面设计规范 V1.0》及 .xd/.rplib 组件源文件（数研院规范）。

## 8. 角色体系（按 BA-02 / 工作指引附录 H）

总部网级：副总经理/总经理（宏观决策、签署授权协议甲方、批准批量授权清单）、经理/高级经理（终审/复核）、主管（发起/归集/制卡/执行授权/系统规则管理）、**数据产权合规管控小组**（合规审查、出具合规意见、风险处置闭环——核心风控角色）、领导小组办公室（最高审议批准）、业务管理部门经理（业务视角审查）。
省/地市：分管领导（审查上报、签授权协议乙方）、部门负责人（数字化=归集初审、业务=提供材料/初审）。基层单位仅作数据执行，不直接使用核心功能。
数据可见范围严格按 RBAC + 行政组织 + 敏感度 + 业务条线 + 权益类型隔离。
